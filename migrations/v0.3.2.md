# Migration notes for v0.3.2

Apply the following changes to consumer repositories before pinning the new SHA.

## Change: caller checkout path renamed to scan_repo

### Who is affected

Consumer repositories that call `full-scan.yml` from cicd-guardrails (directly or via a thin wrapper such as `security.yml`). Forks that vendored `full-scan.yml` with `path: target` must update their copy.

### What changed

The reusable workflow checks out the caller repository to `scan_repo` instead of `target`. All check scripts and `aggregate_risk_summary.sh` receive `scan_repo` as their first argument.

### Why

The file-discovery helper excludes Rust build output via the pattern `*/target/*`. When the entire consumer clone lived under a directory named `target`, every repository file matched that exclude and checks reported empty scan coverage with a false PASS.

### Action required

- Pin `uses: …/full-scan.yml@<SHA>` and `guardrails-ref` to a commit that includes this change.
- If you forked or copied `full-scan.yml`, rename checkout `path: target` to `path: scan_repo` and pass `scan_repo` to check scripts.
- No changes to `.guardrails.yml` are required for this migration.

### Symptom before fix

- CICD-SEC-03 (and other repo-wide checks) show Scan coverage with `found 0` for manifests, workflows, and Dockerfiles despite files existing in the repository.
- Status may be PASS with zero errors and zero warnings.

### Detection

```bash
# In consumer repo: old pin or forked workflow still using target as checkout path
rg 'path: target' .github/workflows
# Ignore job display names that mention pull_request_target (SEC-04), not checkout paths.
```

### Code examples

#### Before

```yaml
- uses: actions/checkout@v5
  with:
    path: target
- run: bash guardrails/scripts/checks/domain/cicd_sec_03.sh target
```

#### After

```yaml
- uses: actions/checkout@v5
  with:
    path: scan_repo
- run: bash guardrails/scripts/checks/domain/cicd_sec_03.sh scan_repo
```
